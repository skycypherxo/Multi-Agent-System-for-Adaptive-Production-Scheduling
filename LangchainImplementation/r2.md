# R2: Complete Working Summary and Run Instructions

## 1) What this system is

This project is a multi-agent production scheduling simulation with Agent-to-Agent (A2A) communication.

Main flow:

JobAgent -> SchedulerAgent -> MachineAgent(s)

The system supports two decision backends:

- Micro mode: local trained instruction policy and micro model path
- GPT mode: local HuggingFace GPT-2 text generation path

It also uses RAG retrieval to provide domain knowledge context for scheduling decisions.

## 2) Core files and their role

- agents.py
  - Main agent logic (JobAgent, SchedulerAgent, MachineAgent)
  - Contains mode switching between micro policy and GPT-2

- a2a/server.py
  - Central communication hub for agents

- a2a/client.py
  - Client used by agents to register, send tasks, poll inbox, update status

  - Factory scheduling knowledge used for retrieval context

- rag_chain.py
  - Standalone LangChain RAG chain implementation
  - Defaults to GPT-2 if no custom LLM is provided

- instruction_policy_model.py
  - Trainable local instruction policy model
  - Learns capability routing + planning priors from dataset

- data/instruction_policy_dataset.jsonl
  - Training dataset for instruction policy

- benchmark_end_to_end_agentic.py
  - End-to-end benchmark script (micro vs gpt)
  - Includes CLI choice: 1 for micro, 2 for GPT-2

- benchmark_llm_routing.py
  - Routing-focused benchmark script

- test_full_system.py
  - Full functional simulation test

## 3) End-to-end execution behavior

### 3.1 Agent communication

1. JobAgent creates jobs and sends them to SchedulerAgent through A2A.
2. SchedulerAgent reads jobs from inbox and assigns a machine.
3. MachineAgent receives assigned job, generates execution plan, marks task complete.
4. Server stores task state transitions: pending -> in-progress -> completed/failed.

### 3.2 Scheduler decision step

For each job, scheduler:

1. Retrieves relevant context from vector store (RAG retrieval)
2. Chooses machine using one of the two modes:
   - Micro instruction policy path
   - GPT-2 generation path with name extraction fallback

### 3.3 Machine planning step

For each assigned task, machine:

1. Builds plan via instruction policy (micro mode) OR generation path (gpt mode)
2. Adds task to machine schedule
3. Updates A2A task status and result payload

---

## 4) What is happening in RAG vs normal mode

RAG retrieval is used in scheduler flow to fetch knowledge snippets relevant to each job.

Then generation/decision differs by mode:

- Micro mode
  - Uses trained instruction policy for decision/planning
  - RAG is used as additional context signal

- GPT mode
  - Uses local GPT-2 generation for decision/planning
  - RAG context is inserted into prompt text

Important: GPT here means local HuggingFace GPT-2 in this repository, not OpenAI API GPT.

## 7) Typical observed behavior

- GPT-2 mode can complete tasks but may misroute some machine assignments depending on output parsing behavior.

## 9) Run instructions (end-to-end)

## 9.1 Terminal 1: start A2A server

From project root:

```powershell
cd .\LangchainImplementation
& ..\.venv\Scripts\Activate.ps1
python -m a2a.server
```

Keep this terminal running.


## 11) Model summary (requested details)

### 11.1 Instruction dataset size

- Dataset file: data/instruction_policy_dataset.jsonl
- Total instruction records: 30
- Target classes: cutting, welding, painting

### 11.2 Model types and parameters

- MicroByteLM (runtime micro model)
  - Type: byte-level bigram MLP (Embedding -> Linear -> Softmax)
  - Parameters: 2304
  - Typical saved weights: artifacts/micro_lm_weights.npz

- GPT-2 (local HuggingFace baseline)
  - Type: decoder-only transformer language model
  - Parameters: 124,439,808
  - Used in generation path when micro mode is off

- InstructionPolicyModel (trained instruction policy)
  - Type: Multinomial Naive Bayes text classifier + learned planning priors
  - Neural parameters: not applicable (non-neural statistical model)
  - Learned artifact: artifacts/instruction_policy_model.json
  - Current learned vocab size: 71 tokens
  - Trained examples in artifact: 30

### 11.3 What RAG does in this system

- RAG retrieves relevant scheduling knowledge snippets from vector store.
- Those snippets are then consumed by either:
  - Instruction policy path (micro mode), or
  - GPT-2 generation path (gpt mode).

### 11.4 Practical conclusion so far

From current end-to-end runs:

- Micro mode: 100% completion, 100% capability match, 100% valid plans, much lower latency (around 7.68s in latest micro run).
- GPT-2 mode: 100% completion, 33.33% capability match, 100% valid plans, much higher latency (around 46.19s in latest gpt run).

Interpretation:

- For this project setup, micro policy is currently better for deterministic routing and speed.
- GPT-2 path still completes tasks, but routing quality is lower under current extraction/parsing behavior.
